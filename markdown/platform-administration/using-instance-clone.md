# Managing Instance Clone

You can use Instance Clone to request, cancel, schedule, or modify clones.

-   **[Cancel a clone](cancel-clone-cac.md#)**  
You can cancel a clone that is scheduled, requested, or in progress until the Node Repoint state. Canceling a clone retains the target instance as-is in its pre-clone state with its original data.
-   **[Handling a large amount of in-progress update sets](clone-update-sets-example.md)**  
You can batch a large group of update sets in the Clone Admin Console.
-   **[Roll back a clone](roll-back-a-clone.md#)**  
Roll back a clone to remove the latest cloning updates on a cloning target if a mistake was made or an error has occurred.
-   **[Schedule recurring clones \(legacy\)](schedule-cloning.md)**  
Schedule recurring clones to keep your cloned instances up to date.
-   **[View the clone history page \(legacy\)](t_ViewCloneHistory.md)**  
You can view the status and history of any instance clone request.
-   **[View clone status \(legacy\)](view-clone-status.md)**  
View the status of a legacy clone.
-   **[Preserving data from target instances during clones](../concept/data-preservation.md)**  
Use data preservers to protect data on the target instance from being overwritten. If you have custom applications, you must also manually preserve unpublished application content.
